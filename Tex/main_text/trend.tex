\subsection{从图级融合到编译期 kernel 划分}

GPU 算子融合的优化单位经历了从算子节点到 tile 与更细计算片段的持续细化。TVM 在张量表达和计算图层执行算子融合与跨算子优化，DNNFusion 根据算子关系和数据流属性生成融合计划，Welder 进一步以 tile graph 描述生产者与消费者之间的数据复用，从而联合调度算子内与算子间访存\cite{chen2018tvm,niu2021dnnfusion,shi2023welder}。FlashAttention 则从算法层重排 Attention 的分块与归约，使中间 attention matrix 保持在片上存储层级，展示了数据流重构与 kernel 边界设计的协同收益\cite{dao2022flashattention,dao2023flashattention2,shah2024flashattention3}。

直观地说，这一阶段要回答的是：一段逻辑计算应被编译成几个 GPU kernel、每个 kernel 覆盖哪些计算片段。Korch 把高层算子拆成更细的代数原语，再用优化方法决定如何把这些原语切分、拼装为最终 kernel 集合，从而允许在原始算子边界之内切开、也允许跨越边界重组\cite{hu2024korch}。ClusterFusion 则把协作范围扩大到 cluster 级集合通信原语，使跨 thread block 的数据交换与同步也进入融合搜索空间\cite{luo2025clusterfusion}。这些工作共同说明：逻辑计算图到 GPU kernel 的映射是一个组合优化问题，收益同时取决于 launch 数量、中间数据物化、片上复用、同步和资源占用。

上述方法主要在代码生成阶段决定“切成几个 kernel、每个装什么”。Megakernel 把问题继续推进到设备端执行：多个算子已经进入同一逻辑任务图之后，依赖由何种运行时机制承载、任务以何种粒度投放、逻辑任务图映射为几个物理 grid，成为新的物理执行计划变量。

\subsection{Persistent Megakernel}

现有 Megakernel 实现大体沿两条路径展开。FlashFormer 这类 whole-model kernel 以固定 cooperative grid 执行一层或多层 Transformer 阶段：面向低 batch 推理，把整模型前向数据流映射到单个 kernel，并通过跨阶段流水缓解权重搬运与计算之间的空隙\cite{nrusimha2025flashformer}；DeepGEMM 的 MegaMoE 则面向专家路由场景，在 SM90 上把 dispatch、专家侧 GEMM、SwiGLU 与 combine 等阶段组织进同一 cooperative kernel，由 grid 内协作完成跨阶段数据交接与同步推进\cite{deepgemm2026,deepgemmPr360}。二者工作分工与同步位置多在编译期确定，执行时依靠 grid-wide 协作推进固定阶段序列，launch 与中间物化开销低，但同步粒度相对较粗。

另一条路径采用常驻 workers 执行设备端 task graph。Hazy Research Megakernels 将算子任务编译为各 SM 的指令队列，worker 根据 event、counter 和 barrier 推进跨算子依赖\cite{hazy2025nobubbles,hazyMegakernels2025}；MPK 进一步提出 SM-level graph 表示，将张量程序降低为细粒度 tasks，并由去中心化的设备内运行时在 persistent megakernel 中调度，论文报告端到端时延改进最高达到 $1.7\times$\cite{cheng2025mpk}。与静态 cooperative 方案相比，task-graph runtime 能够在依赖允许时让不同 SM 独立推进，从而把整 grid 同步细化为 task 或 tile 级事件。

上述实现调度结构不同，却承担相近的系统职责：一次物理 grid 需要在多个阶段间维持任务发现、ready 判断、数据可见性和 worker 前向进展。其关键收益来自消除 launch 与中间物化，并尽可能把局部 readiness 转化为跨算子流水；其关键成本来自设备端调度、软件同步和不同阶段共享同一执行状态。

\subsection{动态 task/event 与自动执行计划生成}

真实推理中的 sequence length、batch size 和 MoE routing 会改变 task 数量及依赖关系。Event Tensor 把事件状态提升为 tensor abstraction，用静态和动态 event update/trigger 表达 shape-dependent 与 data-dependent task graph，并由编译器生成 persistent kernel 的静态或动态调度\cite{eventTensor2026}。Ada-MK 针对固定部署配置执行细粒度 DAG 离线搜索，把运行时分支前移到编译期，同时用异构引擎分别承载 prefill 与 decode\cite{dong2026adamk}。两项工作从不同方向说明，Megakernel 执行计划由 workload shape、依赖结构和硬件资源共同确定。

自动化工具也开始进入 Megakernel 代码生成。AutoMegaKernel 以 schedule IR validator 检查 agent 生成计划的 race、deadlock 和资源约束，并从统一代码生成面向多代 NVIDIA GPU 的 persistent cooperative kernel\cite{jaber2026automegakernel}。这一进展使执行计划搜索具备更大的候选空间，也提高了对可验证依赖表示和可复现实验接口的要求。对物理分区而言，边界决策需要与 task schedule 一同进入候选计划，并通过依赖合法性检查后再进行性能选择。


\subsection{CUDA Graph 与 Programmatic Dependent Launch}

CUDA Graph 把 kernel、memcpy 与同步节点预先实例化为可重复执行的 DAG，从而摊销逐 kernel 的主机提交成本；默认 dependency edge 在上游节点完成后放行下游节点\cite{nvidia2026cudaGraphs}。这一机制降低了多 grid 计划的 host-side cost，并保留每个 grid 独立的资源配置和完成语义，因而常作为物理分区之后的提交与重放载体。

Hopper 及后续架构提供 Programmatic Dependent Launch（PDL）。当 producer 的全部 blocks 分别显式触发 programmatic completion，或在退出时完成隐式触发后，consumer grid 获得提前调度机会；consumer 先执行不读取 producer 输出的独立前缀，并在 dependent read 前调用 \texttt{cudaGridDependencySynchronize()}。驱动对实际并发采用机会式调度，执行计划的正确性由同步语义独立保证\cite{nvidia2026pdl}。CUDA Graph 也可以用 programmatic edge data 表达同一依赖。

CUTLASS 将 PDL 接入 GEMM launch API，其示例主要展示：在已经分立的 producer--consumer GEMM 上，把 consumer 的独立权重预取放到 dependency synchronization 之前，从而使权重访问与 producer 尾部重叠\cite{nvidia2026cutlassDependentLaunch,nvidia2026cutlassExample63}。这一用法证明了 PDL 交接接口的可行性，但决策空间仍较窄——物理边界事先给定，可重叠工作也基本限定为固定模式的权重预取。本课题面向 Megakernel 任务图，把依赖分析、边界选择与 PDL 配置放进同一框架：先判断一条边应保留在同一 grid 内，还是映射为跨 grid 完成语义；再为选定物理边界决定 producer trigger 时机、可提前执行的独立前缀形态，以及预取量与存储层级。

\subsection{现状归纳与研究机会}

现有研究已经形成三个相互衔接的优化层次。图级融合与编译期 kernel 划分决定逻辑计算被切成哪些 kernel；Megakernel 编译器与运行时选择 task 粒度、事件表示和设备端调度；CUDA Graph 与 PDL 优化多个物理 grid 的提交和交接。三个层次分别成熟，却尚未被系统地贯通：逻辑上已经融合的任务图，仍然缺少一套依据依赖形态选择物理执行方式的方法；PDL 虽已证明可在既定边界上重叠独立工作，却尚未与“该不该划边界、何时放行后继、以何种机制调度”一并纳入同一决策过程。

由此形成若干明确的研究机会。第一，建立明确的抽象层次并基于此进行依赖分析，明确计算图中每条相邻算子边的处理范式——例如保留同一 grid 内的细粒度事件同步或映射为跨 grid 的完成语义——并在此候选集合上搜索覆盖全图的边处理方案，使局部边的选择能够组成全局更优的执行计划。第二，对已被判定为跨 grid 的边，可进一步以 PDL 为硬件基础，构建依赖感知的跨边界重叠方案：把 Megakernel 侧的就绪与投放约束、与 CUDA Graph / PDL 侧的触发时机和独立前缀衔接起来，使跨 grid 交接不再停留在既定边界上的固定预取示例，而能够依据依赖形态选择细粒度可重叠工作与放行时机。第三，基于上述抽象建立面向任意计算图的完整框架：从任务拆解、依赖刻画到执行计划生成与运行时落地，并在真实解码负载上检验端到端收益。
